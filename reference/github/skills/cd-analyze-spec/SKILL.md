## 注意

- 技術的な説明（クラス名・メソッド名の羅列）ではなく、業務視点の説明を優先すること
- コードに書かれていないことは推測であることを明示すること
- 説明は設計書の代替として使えるレベルの粒度を目指すこと

## 必要な情報（未指定なら質問すること。対話できない場合は妥当な仮定を置き、仮定した内容を出力に明記すること）

- 対象のクラス名またはメソッド名
- 対象の業務領域・機能名（わかる範囲で。コードの解釈精度を上げるために活用する）

> コーディング規約は `copilot-instructions.md` に従う。規約と既存コードのパターンが矛盾する場合は既存コードを優先すること。

指定されたクラス・メソッドについて、業務ロジックを日本語で説明してください。

## 説明内容

1. **処理の目的**: このクラス・メソッドが業務上で何をするものか（1〜2行で）
2. **業務ロジックの説明**: 処理の流れを業務担当者にも伝わる言葉で説明（技術用語は補足として添える）
3. **条件分岐の業務的な意味**: if/switch 等の分岐を「〇〇の場合は〜、それ以外は〜」の形式で解釈
4. **入力・出力の意味**: 引数と戻り値が業務上で何を表しているか
5. **不明・不確かな箇所の明示**: コードだけでは業務的な意図を断定できない箇所は「推測」として明示すること

## ファイル出力

説明結果は以下のルールに従ってファイルに保存してください。

### 出力先

```
docs/analysis/
└── YYYY-MM-DD_NN/
    └── spec-recovery.md
```

- `YYYY-MM-DD` は実行日の日付
- `NN` は同一日内の2桁連番（既存ディレクトリを確認し、最大の連番 +1 を採番。初回は `01`）
- ディレクトリが存在しない場合は作成すること
- 同一セッションで複数クラスを分析した場合は、同一ファイルに追記すること

### ファイルの冒頭

```markdown
# 仕様復元レポート — YYYY-MM-DD_NN

## 分析対象
（対象クラス名・メソッド名、業務領域・機能名を記載）
```

その後、「説明内容」の1〜5をそのまま続けて記載してください。
