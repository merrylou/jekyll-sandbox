## 注意

- フレームワーク（Spring, Hibernate 等）内のスタックフレームは原因特定の補助として扱い、自プロジェクトのコードを優先的に調査すること
- 「このメソッドが原因」と断定せず、「このメソッドで〇〇が起きうる」と根拠を示して提示すること
- ログから読み取れる情報の範囲で判断し、確認が必要な情報は「次のアクション」として明示すること
- 修正案は求められた場合のみ提示すること（解析と修正を混在させない）

## 必要な情報（未指定なら質問すること。対話できない場合は妥当な仮定を置き、仮定した内容を出力に明記すること）

- エラーログまたはスタックトレース（コピー＆ペースト）
- エラーが発生した環境（本番 / ステージング / ローカル等）
- エラーが発生する操作・画面・機能（わかる範囲で）
---


> コーディング規約は `copilot-instructions.md` に従う。規約と既存コードのパターンが矛盾する場合は既存コードを優先すること。

以下のエラーログ・スタックトレースを解析してください。

## 解析内容

1. **エラーの種別と発生箇所**: 例外クラス名・エラーメッセージから問題の種別を特定し、スタックトレース上の自プロジェクトコードの行を特定する
2. **コードとの照合**: スタックトレースに現れるクラス・メソッドをコードベースで確認し、該当箇所の処理内容を説明する
3. **原因候補のリストアップ**: コードを読んで考えられる原因を列挙し、各候補の根拠と可能性（高/中/低）を示す
4. **再現に必要な条件**: ログから読み取れる入力値・状態・実行タイミングなど、エラーが発生した状況の整理
5. **次のアクション**: 原因を絞り込むために確認すべきこと（追加ログ、DBレコード、実行時の入力値など）

## 手順

1. スタックトレースのうち、フレームワーク・ライブラリ部分と自プロジェクト部分を分離する
2. 自プロジェクト部分のクラス・メソッドをプロジェクト内を検索し、実際のコードと照合する
3. エラーメッセージ・行番号・例外の種類を組み合わせて原因を推定する
4. 複数の原因候補がある場合は可能性の高い順に提示する

## ファイル出力

解析結果は以下のルールに従ってファイルに保存してください。

### 出力先

```
docs/analysis/
└── YYYY-MM-DD_NN/
    └── error-log-analysis.md
```

- `YYYY-MM-DD` は実行日の日付
- `NN` は同一日内の2桁連番（既存ディレクトリを確認し、最大の連番 +1 を採番。初回は `01`）
- ディレクトリが存在しない場合は作成すること
- 同一セッションで複数のエラーを解析した場合は同一ファイルに追記すること

### ファイルの冒頭

```markdown
# エラーログ解析レポート — YYYY-MM-DD_NN

## 解析対象
（エラーが発生した環境・機能・発生日時）

## エラーログ（貼り付けたログ全文）
（ここに貼り付けたログをそのまま引用）
```

その後、「解析内容」の1〜5をそのまま続けて記載してください。
