## 注意

- クラス名・メソッド名は実際にコードベースに存在するもののみを使うこと
- 「このメソッドが原因」と断定せず、「このメソッドで〇〇が起きうる」と根拠を示して提示すること
- 修正案は求められた場合のみ提示すること（調査と修正を混在させない）
- 調査の結果、不具合が既に修正済みと判断した場合は、修正コミットの特定と修正の妥当性評価を行い、その旨を報告すること。修正済みの場合でもテストで再発防止されているかを確認すること

## 必要な情報（未指定なら質問すること。対話できない場合は妥当な仮定を置き、仮定した内容を出力に明記すること）

- 不具合が発生する画面・機能・操作手順
- エラーメッセージまたは誤った動作の内容
- 関連するパッケージパスまたはクラス名（わかる範囲で）
---


> コーディング規約は `copilot-instructions.md` に従う。規約と既存コードのパターンが矛盾する場合は既存コードを優先すること。

以下の不具合について調査してください。

## 調査内容

1. **処理フローの追跡**: 指定された画面・機能のリクエスト受信からDB操作・レスポンス返却までのコールスタック
2. **エラー発生候補箇所**: バリデーション・例外処理・条件分岐のうち、不具合が発生しうる箇所
3. **関連データの流れ**: 問題のある入力値やDBレコードがどのクラス・メソッドで生成・参照・加工されるか
4. **横展開リスク**: 同じロジックパターンが他の機能にも存在するか（類似バグの可能性）。例: 大文字小文字を区別する検索が原因であれば、他の検索機能にも同様の問題がないかを確認する
5. **再現条件の整理**: 不具合を再現するために必要な前提条件（データ状態、環境設定、操作手順）を整理する

## 手順

1. 不具合の発生箇所（画面・機能・エラーメッセージ）を確認する
2. エントリーポイントとなるクラスを特定する
3. 関連クラスをプロジェクト内を検索して追跡し、コールスタックを構築する
4. 疑わしい箇所をリストアップし、根拠とともに提示する

## ファイル出力

調査結果は以下のルールに従ってファイルに保存してください。

### 出力先

```
docs/analysis/
└── YYYY-MM-DD_NN/
    └── bug-investigation.md
```

- `YYYY-MM-DD` は実行日の日付
- `NN` は同一日内の2桁連番（既存ディレクトリを確認し、最大の連番 +1 を採番。初回は `01`）
- ディレクトリが存在しない場合は作成すること

### ファイルの冒頭

```markdown
# 不具合調査レポート — YYYY-MM-DD_NN

## 調査対象
（不具合が発生する画面・機能・操作手順を記載）

## 報告されたエラー内容
（エラーメッセージまたは誤った動作の内容を記載）
```

その後、「調査内容」の1〜5をそのまま続けて記載してください。
