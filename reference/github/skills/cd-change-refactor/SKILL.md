> コーディング規約は `copilot-instructions.md` に従う。規約と既存コードのパターンが矛盾する場合は既存コードを優先すること。

以下の対象コードをリファクタリングしてください。

## 手順

1. 対象パッケージ配下の全クラスを読み取る
2. 指定された目的の観点で改善箇所を特定する
3. リファクタリングを実施する
4. 変更サマリー（変更箇所・変更理由）を出力する

## 制約

- 公開メソッドのシグネチャは変更しない
- 既存テストを壊さない
- 変更理由が説明できない変更は行わない

## 必要な情報（未指定なら質問すること。対話できない場合は妥当な仮定を置き、仮定した内容を出力に明記すること）

- 対象パッケージまたはクラス群のパス
- リファクタリングの目的（可読性向上、重複除去、パターン適用等）
- 変更してはいけない箇所の制約（あれば）

## 改善観点の例（目的に応じて参照）

リファクタリングの目的が曖昧な場合、以下の観点から改善箇所を特定する:

- **未使用コード**: 未使用のimport文、変数、メソッド、クラス
- **命名の一貫性**: 同じ概念に対して異なる命名パターンが混在していないか
- **冗長なコード**: 不要なローカル変数、冗長な条件分岐、不要なキャスト
- **エラーメッセージの体裁**: 同種のメッセージの形式・末尾処理が統一されているか
- **コーディングスタイルの統一**: メソッドチェーン vs 中間変数、`this.`プレフィックスの有無等

## 注意

- アーキテクチャ上意図的に存在する類似パターン（例: 各Controllerに共通するCRUD処理）と、排除すべき不要な重複を区別すること。判断が難しい場合はその旨を明記し、重複除去を強行しない
- 変更結果のサマリーでは、各変更について「なぜこの変更が必要か」を説明すること
