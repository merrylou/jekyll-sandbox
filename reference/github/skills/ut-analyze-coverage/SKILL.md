> コーディング規約は `copilot-instructions.md` に従う。規約と既存コードのパターンが矛盾する場合は既存コードを優先すること。

指定されたファイルを読み取り、テストカバレッジのギャップを分析してください。

## 必要な情報（未指定なら質問すること。対話できない場合は妥当な仮定を置き、仮定した内容を出力に明記すること）

- テスト対象のファイルパス
- 既存テストのファイルパス
- カバレッジ目標（例: 命令網羅80%、分岐網羅80%）
- JaCoCoレポート（HTMLまたはXML）のパス、またはカバレッジ未達行のコピー貼り付け（あれば）
- 重点的にカバーしたい観点（例外系、分岐条件、null処理等。あれば）

## カバレッジ分析の方針

**JaCoCoレポートがある場合:**
- レポートのカバレッジ未達行・分岐を基準に分析する
- 未カバーの行番号をテスト対象コードと照合し、不足しているテストケースを特定する

**レポートがない場合（コード静的解析）:**
- テスト対象クラスの全分岐（`if`/`else`/`switch`/三項演算子）・例外パス・null チェックを洗い出す
- 既存テストのアサーション・Mockの設定から、どの分岐がカバーされているかを判定する

## 手順

1. テスト対象クラスの分岐・条件を網羅的に洗い出す
2. 既存テストとの照合で未カバー箇所を特定する
3. 未カバー箇所の一覧を出力する（行番号・分岐条件・未カバーの理由）
4. 不足分のテストメソッドを既存テストファイルに追記する形で生成する
5. 追加したテストを実行し、全件パスすることを確認する
6. 追加したテストケース一覧（メソッド名・観点・カバーする分岐）を表形式で出力する

## 制約

- 既存テストのスタイル・命名規則を維持すること
- 既存テストの修正は行わず、追記のみとする

## 注意

- JaCoCoレポートなしでのコード静的解析は、明示的な分岐（if/else/switch/三項演算子）の検出には有効だが、フレームワーク内部の分岐やAOP的な処理はカバーできない。静的解析での分析後、テスト追加→JaCoCo計測で残りのギャップを確認する2段階アプローチを推奨する
