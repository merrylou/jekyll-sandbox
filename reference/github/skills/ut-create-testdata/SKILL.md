> コーディング規約は `copilot-instructions.md` に従う。規約と既存コードのパターンが矛盾する場合は既存コードを優先すること。

指定されたテーブル定義またはエンティティクラスを読み取り、テストデータを生成してください。

## 必要な情報（未指定なら質問すること。対話できない場合は妥当な仮定を置き、仮定した内容を出力に明記すること）

- テーブル定義（DDL）またはEntityクラスのファイルパス
- 出力形式（SQL INSERT文 / JSON / Javaテストフィクスチャ / `@Sql`アノテーション用SQLファイル）
- 関連テーブルの定義（FK制約がある場合）
- テスト対象のユースケース・シナリオ（あれば。より適切なデータを生成するために使用）

## 生成条件

- **正常系**: 代表的なパターンを網羅（3〜5件）
- **境界値**: 文字列カラムの最大長、数値カラムの上限/下限、日付の境界
- **異常系**: NULL許容カラムのNULL、特殊文字（`'`、`"`、`<`、`>`）を含むデータ
- FK制約・一意制約・NOT NULL制約を満たすこと
- DDL上の制約（NOT NULL、文字長等）とEntityクラスのバリデーション制約（`@NotBlank`、`@Size`等）が異なる場合がある。DB層テスト用データはDDL制約に従い、アプリ層テスト用データはEntityバリデーション制約にも従うこと。どちらの制約に基づくかをデータのコメントに明記すること
- メールアドレスは `test+xxx@example.com` 形式を使用
- パスワードはダミーハッシュ値（例: `$2a$10$dummyhashvalue...`）を使用
- 日付・日時はテスト意図が明確な値を使用（例: 境界日付、過去/未来の代表値）

## Spring Boot 利用時の補足

- `@Sql`アノテーション用として出力する場合は、`INSERT`文のみのSQLファイルを生成し、クリーンアップ用の`DELETE`文も別途生成する
- `@DataJpaTest` や `@SpringBootTest` との組み合わせを考慮し、トランザクション制御のコメントを付与する

## 出力形式

生成後に、以下を出力すること：
- 生成したデータの一覧（件数・各レコードの意図）
- FK制約を満たすために必要な親テーブルのデータがある場合はその挿入順序
