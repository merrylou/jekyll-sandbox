## 注意

- クラス名・メソッド名は実際にコードベースに存在するもののみを使うこと
- 影響なしと判断した箇所についても、その根拠を簡潔に示すこと
- 修正の実施は求められた場合のみ行うこと（評価と実装を混在させない）

## 必要な情報（未指定なら質問すること。対話できない場合は妥当な仮定を置き、仮定した内容を出力に明記すること）

- 変更対象のクラス名またはメソッド名
- 変更の概要（何をどう変えるか）
- 影響確認の対象範囲（パッケージパス、あれば絞り込みに活用する）

> コーディング規約は `copilot-instructions.md` に従う。規約と既存コードのパターンが矛盾する場合は既存コードを優先すること。

以下の変更を実施する前の影響範囲を評価してください。

## 評価内容

1. **呼び出し元の一覧**: 変更対象のクラス・メソッドを利用している呼び出し元（Controller / Service / バッチ / テンプレート・ビュー等）
2. **DB依存の確認**: 変更対象が読み書きするテーブル・カラムと、それを共有している他の機能。スキーマファイルが複数DB方言で存在する場合は全方言を確認すること
3. **View・テンプレート層への影響**: 変更対象のフィールドやメソッドを参照しているテンプレートファイル（Thymeleaf / JSP 等）の有無
4. **リソースファイルへの影響**: i18n（メッセージプロパティ）、設定ファイル等への影響
5. **共通部品への影響**: 変更対象が共通クラス・ユーティリティの場合、影響を受ける全機能の範囲
6. **テストカバレッジの確認**: 変更対象に対応する既存テストの有無と、修正が必要になりそうなテストの一覧
7. **データ移行の考慮**: DBスキーマ変更を伴う場合、既存データへの影響（NOT NULLカラム追加時のデフォルト値、データ型変更時の既存データ互換性等）
8. **変更リスク評価**: 影響範囲の広さ・テストの充実度をもとに、高/中/低でリスクを判定

### リスク判定の目安

| リスク | 基準 |
|--------|------|
| 高 | 影響ファイル10以上、またはDB スキーマ変更あり、またはテストカバレッジが低い箇所への変更 |
| 中 | 影響ファイル3〜9、テストカバレッジが一定ある、変更パターンが定型的 |
| 低 | 影響ファイル2以下、テストで十分カバー済み、コードスタイル変更等 |

## 手順

1. 変更対象のクラス・メソッドをプロジェクト内を検索し、呼び出し元を列挙する
2. SQL・Repositoryクラスから関連テーブルを特定する
3. 影響範囲を整理し、変更時に注意すべき点をリストアップする
4. リスク評価の根拠を明示して提示する

## ファイル出力

評価結果は以下のルールに従ってファイルに保存してください。

### 出力先

```
docs/analysis/
└── YYYY-MM-DD_NN/
    └── impact-analysis.md
```

- `YYYY-MM-DD` は実行日の日付
- `NN` は同一日内の2桁連番（既存ディレクトリを確認し、最大の連番 +1 を採番。初回は `01`）
- ディレクトリが存在しない場合は作成すること

### ファイルの冒頭

```markdown
# 影響範囲評価レポート — YYYY-MM-DD_NN

## 変更対象
（変更対象のクラス名・メソッド名を記載）

## 変更概要
（何をどう変えるかを記載）
```

その後、「評価内容」の1〜8をそのまま続けて記載してください。
