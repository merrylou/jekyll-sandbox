> コーディング規約は `copilot-instructions.md` に従う。規約と既存コードのパターンが矛盾する場合は既存コードを優先すること。

以下の変更仕様に基づいて、既存コードを改修してください。

## 手順

1. 対象ファイルと関連ファイルを読み取る。関連ファイルの特定方法:
   - 対象ファイルの `import` 先クラス
   - 対象クラスを参照している呼び出し元（プロジェクト内検索）
   - 対象エンティティに対応するテンプレート・ビューファイル
   - 対象エンティティのDDL（スキーマファイル）
   - 対象機能に関連するテストファイル
2. 既存のコーディングスタイル・例外処理パターンを把握する
3. 変更仕様に基づいてコードを改修する
4. 影響範囲（呼び出し元、関連するService/Repository）をプロジェクト内を検索して確認する
5. 影響を受けるファイルがあれば、必要な修正を提案する
6. 関連する既存テストの修正も提案する

## 注意

- 手順1〜2の調査の結果、変更が既に実装済みまたは不要と判断した場合は、その根拠を示して報告し、不要な変更を実施しないこと
- 影響範囲の確認では、Javaクラスだけでなくテンプレート（Thymeleaf/JSP等）、設定ファイル、テストファイルも対象に含めること

## 必要な情報（未指定なら質問すること。対話できない場合は妥当な仮定を置き、仮定した内容を出力に明記すること）

- 改修対象のファイルパス
- 変更内容の詳細（何をどう変えるか）
- 影響確認の範囲（関連パッケージパス）
- 設計書がなくTODOコメントのみの場合は、TODOコメントと周辺コードから仕様を推測し、推測箇所を明示すること
