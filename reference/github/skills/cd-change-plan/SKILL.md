## ファイル出力

変更計画は以下のルールに従ってファイルに保存してください。

### 出力先

```
docs/analysis/
└── YYYY-MM-DD_NN/
    └── change-plan.md
```

- `YYYY-MM-DD` は実行日の日付
- `NN` は同一日内の2桁連番（既存ディレクトリを確認し、最大の連番 +1 を採番。初回は `01`）
- ディレクトリが存在しない場合は作成すること


## 注意

- 計画フェーズではコードの変更を行わないこと
- 「どちらでもよい」とは言わず、現場での実施容易性を優先して推奨案を1つ示すこと
- 既存のコーディングパターンと異なるアプローチを推奨する場合は、その理由を明示すること

## 必要な情報（未指定なら質問すること。対話できない場合は妥当な仮定を置き、仮定した内容を出力に明記すること）

- 修正対象のクラス名またはファイルパス
- 変更要件の内容（何をどう変えたいか）
- 制約条件（変更できない箇所、考慮が必要な非機能要件など）
---


> コーディング規約は `copilot-instructions.md` に従う。規約と既存コードのパターンが矛盾する場合は既存コードを優先すること。

以下の変更に対する実装アプローチを比較・提案してください。

**重要: まず変更計画のみを提示し、実装の承認を得てから実装に進むこと。**

## 計画フェーズ（承認前に実施）

### 1. 現状の確認

- 修正対象のクラス・メソッドの現在の実装を確認する
- 呼び出し元・呼び出し先の依存関係を把握する
- 既存テストの有無と内容を確認する

### 2. アプローチの比較

以下の観点で2〜3案を比較してください。

| 観点 | 説明 |
|------|------|
| 実装コスト | コード変更の規模・複雑さ |
| 影響範囲 | 変更が波及するクラス・テストの数 |
| 既存設計との整合性 | 現在のアーキテクチャパターンとの一貫性 |
| テスト容易性 | 変更後のテストのしやすさ |
| リスク | デグレやバグを生むリスクの高低 |
| 要件充足度 | 変更要件をどの程度満たすか（完全/部分的/段階対応） |

### 3. 推奨案の提示

上記比較を踏まえて推奨案を1つ選び、選定理由を明示してください。

### 4. 変更計画のサマリ

推奨案に基づいて、以下を提示してください。

- 修正するファイル・クラス・メソッドの一覧
- 各ファイルへの変更概要（1〜2行）
- 修正の推奨実施順序
- 既存テストの修正・追加が必要な箇所


## ファイル出力

変更計画は以下のルールに従ってファイルに保存してください。

### 出力先

```
docs/analysis/
└── YYYY-MM-DD_NN/
```

## 実装フェーズ（承認後に実施）

計画の承認を受けた後、推奨案に従って実装を開始してください。

実装時は `cd-change-existing` スキルの手順に準拠してください。
