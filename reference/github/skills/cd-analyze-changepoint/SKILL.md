- 変更の種別（機能追加 / 仕様変更 / 廃止）
- 対象機能・画面名（わかる範囲で。検索精度を上げるために活用する）

以下の変更要件を実現するために修正が必要なクラス・メソッドを特定してください。

## 調査内容

1. **エントリーポイントの特定**: 変更要件に関係する画面・API・バッチのエントリーポイントとなるクラスを特定する
2. **修正が必要なクラス・メソッドの一覧**: 要件を実現するために変更・追加が必要なクラス・メソッドを列挙する
3. **修正理由の明示**: 各クラス・メソッドをなぜ修正する必要があるかを簡潔に説明する
4. **新規作成が必要なもの**: 既存コードの修正では対応できず、新たにクラス・メソッドを追加する必要がある箇所
5. **調査の推奨順序**: どの順番で修正を進めると安全か（依存関係を考慮した推奨順）

## 手順

1. 変更要件のキーワード（機能名・画面名・業務用語）からプロジェクト内で関連クラスを検索する
2. エントリーポイントから呼び出し先を追跡し、要件に関与するクラスを特定する
3. 要件の変更内容（追加・変更・削除）に照らして、各クラスの修正要否を判定する
4. 修正対象を「必須」「影響を受けて変更が必要」「変更不要だが確認が必要」の3段階で分類する

## ファイル出力

調査結果は以下のルールに従ってファイルに保存してください。

### 出力先

```
docs/analysis/
└── YYYY-MM-DD_NN/
    └── change-point-analysis.md
```

- `YYYY-MM-DD` は実行日の日付
- `NN` は同一日内の2桁連番（既存ディレクトリを確認し、最大の連番 +1 を採番。初回は `01`）
- ディレクトリが存在しない場合は作成すること

### ファイルの冒頭

```markdown
# 修正箇所特定レポート — YYYY-MM-DD_NN

## 変更要件の概要
（入力された要件を1〜3行で要約）

## 前提・スコープ
（調査対象とした範囲、除外した範囲）
```

その後、「調査内容」の1〜5をそのまま続けて記載してください。
