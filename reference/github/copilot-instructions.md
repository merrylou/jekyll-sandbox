# プロジェクト共通ルール

## 対象技術スタック

| 区分 | 内容 |
|---|---|
| 言語 | Java |
| フレームワーク | Spring Boot |
| レイヤー構成 | Controller → Service → Repository |
| テスト | JUnit5 + Mockito + AssertJ |

## コーディング規約

### 共通

- 既存コードのスタイル・命名規則・パッケージ構成に準拠すること。独自のスタイルを持ち込まない
- **既存パターン優先の適用基準**: 既存ファイルの修正時は既存のスタイルに合わせる。新規ファイルの作成時は同パッケージ内の既存ファイルのスタイルを参照し、該当がなければ本規約に従う
- 既存の例外体系・エラーハンドリングパターンに合わせる
- 公開メソッドのシグネチャ（メソッド名・引数・戻り値）を指示なく変更しない
- 存在しないライブラリ・クラス・メソッド・APIを使用しないこと

### Spring Boot 固有

- **パッケージ構成**: 機能/ドメイン単位で切る（例: `com.example.app.order`）。レイヤー単位のパッケージ分割は避ける
- **DI**: コンストラクタインジェクションを使用する。フィールドは `private final` で宣言する（`@Autowired` フィールドインジェクション禁止）
- **Web層**: JPAエンティティをAPIに直接公開しない。DTO でマッピングすること
- **バリデーション**: リクエスト DTO には `@Valid` を付与し、フィールドに Bean Validation アノテーション（`@NotNull` / `@Size` 等）を使用する
- **トランザクション**: `@Transactional` は Service メソッド単位で付与する（Controller・Repository には付与しない）
- **ログ**: SLF4J を使用し、`logger.info("Processing {}...", id)` のパラメータ化形式で記述する（文字列連結禁止）

## セキュリティポリシー

- セキュリティ関連の実装（認証・認可・暗号化・入力検証等）を独自に追加しないこと
- 機密情報（パスワード・APIキー・トークン等）をハードコーディングしない。`application.yml` の環境変数参照 `${ENV_VAR}` を使用すること
- ログに個人情報・機密情報を出力しない

## データ取扱

- 本番データ・顧客情報・機密情報をコンテキストに含めない
- テストデータに実在の個人情報に似たデータを使用しない。架空のデータ（`test+xxx@example.com` 等）を使用すること
- OSS ライセンスに抵触する可能性のあるコードをそのまま使用しない
