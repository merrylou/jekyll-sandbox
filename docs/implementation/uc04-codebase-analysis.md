# UC04: 既存コードベースの理解

## 目的

プロジェクト参画時やアサイン変更時に、既存コードの構造・設計意図・依存関係をAIを活用して素早く把握する。
コードリーディングにかかる立ち上がり期間を短縮し、早期に開発に貢献できる状態を目指す。

## 利用タイミング

- 新規プロジェクトに参画し、コードベースの全体像を把握したい時
- 担当モジュールが変更され、新たな領域のコードを理解する必要がある時
- 不具合調査で馴染みのないコード箇所を読み解く必要がある時

## Input / Output

| 区分 | 内容 | 形式 |
|------|------|------|
| Input | 理解したいコード範囲 | パッケージ名 / クラス名 / 機能名 |
| Input | 理解したい観点 | 全体構造 / 処理フロー / 依存関係等 |
| Output | アーキテクチャ概要 | レイヤー構成・主要コンポーネントの説明 |
| Output | クラスの責務サマリー | クラスごとの役割と関係性 |
| Output | 処理フロー | リクエストからレスポンスまでの流れ |

## 使い方

> **Copilotユーザー:** スラッシュコマンド `/cd-analyze-codebase` で実行できます。以下はコマンドの内容を理解・カスタマイズするための参考情報です。

### パターンA: Chat（推奨）

コードベースの理解はChatモードでの対話が最も効果的。

```
{{対象パッケージパス}} について教えてください。

1. このパッケージのアーキテクチャ概要（レイヤー構成、主要コンポーネント）
2. 主要クラスそれぞれの責務（1クラス1-2行で）
3. {{主要機能名}} の処理フロー（リクエスト受信からレスポンス返却まで）
4. 外部依存（他パッケージ、外部API、DB等）の一覧

理解の目的: {{目的}}
```

**カスタマイズのポイント:**

- `{{対象パッケージパス}}`: 理解したい範囲（例: `src/main/java/com/example/app/order/`）
- `{{主要機能名}}`: 特に理解したい機能のフロー（例: 「注文確定処理」）
- `{{目的}}`: 理解の背景を伝えると回答の焦点が定まる。以下のように具体的に記述する
  - 「このモジュールに新機能を追加する予定のため、拡張ポイントを知りたい」
  - 「不具合の原因を調査中で、データの流れを把握したい」
  - 「初参画でコードベース全体の概要を掴みたい」

### パターンB: Agent Mode（代替）

より広範囲の分析や成果物出力が必要な場合に使用する。

```
{{対象パッケージパス}} を分析し、以下を出力してください。

1. アーキテクチャ概要（Mermaidのクラス図で主要クラスの関係を図示）
2. {{主要機能名}} の処理シーケンス（MermaidのシーケンスDで図示）
3. 設計上の注意点・制約事項

出力形式: Markdownファイル
```

## 人間が確認すべきこと

- [ ] AIの説明が実際のコードと一致しているか（存在しないクラスやメソッドへの言及がないか）
- [ ] 古いコメントや非推奨コードに基づく誤った説明になっていないか
- [ ] 重要な暗黙知（ワークアラウンド、運用上の注意点等）が抜けていないかチームメンバーに確認する
- [ ] AIの説明だけで理解した気にならず、実際にコードを読んで裏付けを取る

## よくある落とし穴

1. **存在しないクラス・メソッドへの言及（ハルシネーション）**: AIが実際には存在しないクラスやメソッドを説明に含めることがある。AIの説明に登場するクラス名・メソッド名は、実際にコードベース内に存在するか確認する。
2. **AIの説明を設計判断の根拠にする危険**: AIの説明はあくまで理解の補助であり、設計判断の根拠にしてはならない。設計意図の確認は、コードのコメント・コミット履歴・設計ドキュメント・チームメンバーへのヒアリングで裏付けを取る。
3. **大規模コードベースでの参照範囲の限界**: AIが一度に参照できる範囲にはトークン数の制限がある。質問の範囲を絞る（パッケージ単位、機能単位）ことで、より正確な回答を得られる。
