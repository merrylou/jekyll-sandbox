# UC07: エラーログ解析

## 目的

エラーログ・スタックトレースからコードベースと照合して問題箇所を特定し、原因候補を絞り込む。
障害対応や不具合調査の初動を迅速化する。

## 利用タイミング

- 本番・ステージング環境でエラーが発生し、原因調査が必要なとき
- スタックトレースを読んでも該当箇所の処理内容が把握できないとき
- 複数の原因候補を整理して優先順位をつけたいとき

## Input / Output

| 区分 | 内容 | 形式 |
|------|------|------|
| Input | エラーログまたはスタックトレース | コピー＆ペースト |
| Input | 発生環境 | 本番 / ステージング / ローカル等 |
| Input | 発生時の操作・画面・機能（任意） | 再現条件の補助情報 |
| Output | エラーの種別と発生箇所 | 例外クラス名・該当コード行 |
| Output | 原因候補リスト | 可能性（高/中/低）付き |
| Output | 次のアクション | 原因を絞り込むための確認事項 |

## 使い方

> **Copilotユーザー:** スラッシュコマンド `/cd-analyze-errorlog` で実行できます。以下はコマンドの内容を理解・カスタマイズするための参考情報です。

### パターンA: Agent Mode（推奨）

1. `@implementer` エージェントを呼び出す
2. `/cd-analyze-errorlog` プロンプトを実行する
3. エラーログを貼り付け、発生環境・操作を伝える
4. 出力された原因候補と次のアクションを確認する

### パターンB: Chat で段階的に調査

- スタックトレースを貼り付けて「自プロジェクトのコードでどこが原因か」と質問する
- 特定のメソッドの処理内容を確認し、エラー条件を絞り込む

## 人間が確認すべきこと

- [ ] AIが提示した原因候補の根拠が妥当か確認する
- [ ] スタックトレースに現れない間接的な原因（データ不整合、タイミング問題等）を考慮する
- [ ] 「次のアクション」に基づいて追加調査を実施する
- [ ] 原因が特定できたら、修正は別のUC（UC02等）で実施する

## よくある落とし穴

1. **AIの原因推定を検証せずに修正に着手する**: AIは「こういう原因が考えられる」と提示するが、それが真因とは限らない。必ず追加調査で裏付けを取ってから修正すること。
2. **フレームワーク内部のスタックフレームに惑わされる**: Spring・Hibernate等のフレームワーク内部のスタックは量が多いが、原因は多くの場合自プロジェクトのコードにある。AIにもフレームワーク部分と自プロジェクト部分の区別を意識させること。
3. **解析と修正を同時に進める**: このUCはエラーの「原因特定」に専念するもの。原因が確定してから修正（UC02）に進むことで、誤った箇所を修正するリスクを避ける。
