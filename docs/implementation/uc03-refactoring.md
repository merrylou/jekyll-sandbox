# UC03: リファクタリング

## 目的

複数ファイルにまたがるコード改善（重複除去、デザインパターン適用、命名改善等）をAgent Modeで効率的に実施する。
手作業では見落としやすい重複コードや一貫性の欠如をAIで網羅的に検出・修正する。

## 利用タイミング

- 設計レビューやコードレビューでリファクタリングが決定した時
- 技術的負債の解消タスク（スプリントのバッファ作業等）に着手する時
- 機能追加前に、対象コードの構造を整理しておきたい時

## Input / Output

| 区分 | 内容 | 形式 |
|------|------|------|
| Input | リファクタリング対象のコード | パッケージ or クラス群 |
| Input | 改善の目的・方針 | テキスト（具体的なゴール） |
| Output | 改善後のコード | 変更されたソースファイル |
| Output | 変更サマリー | 変更理由と内容の一覧 |

## 使い方

> **Copilotユーザー:** スラッシュコマンド `/cd-change-refactor` で実行できます。以下はコマンドの内容を理解・カスタマイズするための参考情報です。

### パターンA: Agent Mode（推奨）

```
以下の対象コードをリファクタリングしてください。

対象: {{対象パッケージ or クラス群のパス}}
目的: {{リファクタリングの目的}}

手順:
1. 対象パッケージ配下の全クラスを読み取る
2. {{目的}}の観点で改善箇所を特定する
3. リファクタリングを実施する
4. 変更サマリー（変更箇所・変更理由）を出力する

制約:
- 公開メソッドのシグネチャ（メソッド名・引数・戻り値）は変更しない
- 既存テストを壊さない
- {{変更制約}}
- 変更理由が説明できない変更は行わない
```

**カスタマイズのポイント:**

- `{{対象パッケージ or クラス群のパス}}`: リファクタリング範囲（例: `src/main/java/com/example/app/common/util/`）
- `{{リファクタリングの目的}}`: 具体的な改善ゴール。以下のように明示する
  - 可読性向上: 長大メソッドの分割、変数名の改善
  - 重複除去: 複数クラスに散在する同一ロジックの共通化
  - パターン適用: Strategy、Factory等のデザインパターンの導入
- `{{変更制約}}`: プロジェクト固有の制約（例: 「外部API呼び出し箇所は変更しない」）

### パターンB: Chat / Inline（代替）

単一クラス内のリファクタリングにはChatモードで十分。

```
{{対象ファイルパス}} のリファクタリングを提案してください。
目的: {{目的}}
公開メソッドのシグネチャは変更しないでください。
```

## 人間が確認すべきこと

- [ ] 公開メソッドのシグネチャ（外部インターフェース）が変わっていないか
- [ ] 既存テストが全てパスするか（`mvn test` で確認）
- [ ] 変更理由が妥当か（過剰な抽象化・汎用化になっていないか）
- [ ] チームの設計方針・アーキテクチャ規約と整合しているか

## よくある落とし穴

1. **過剰な抽象化・汎用化**: AIは汎用的な設計を好む傾向があり、現在の要件には不要な抽象レイヤーを追加しがちである。「現時点で必要か」をチームの設計方針で判断し、YAGNI原則を適用する。
2. **大量変更によるレビュー不能**: 一度に多数のファイルを変更するとレビューが困難になる。PRは「挙動を変えないリファクタリング」と「挙動変更」を分け、1PRの変更ファイル数を目安10ファイル以内に保つ。
3. **リファクタリングに紛れた挙動変更**: 整理のつもりが実行順序や条件分岐の挙動を変えてしまうことがある。リファクタリング前後で既存テストをパスすることを必ず確認し、カバレッジが不足している箇所はテストを先に追加する。
