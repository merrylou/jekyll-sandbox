# UC02: 既存コードの改修

## 目的

既存機能への項目追加・ロジック変更・API仕様変更等、改修案件での日常的な変更作業をAIで効率化する。
変更箇所の特定から影響範囲の確認まで、改修の一連の流れをAIが支援する。

## 利用タイミング

- 改修対象の設計変更（項目追加、ロジック変更、API仕様変更等）が確定した時点
- 既存コードへの変更箇所が特定でき、改修内容が明確になった段階

## Input / Output

| 区分 | 内容 | 形式 |
|------|------|------|
| Input | 変更仕様（何をどう変えるか） | 設計変更資料 or テキスト |
| Input | 改修対象の既存コード | プロジェクト内のソースコード |
| Output | 変更後のコード | 差分が明確な形式 |
| Output | 影響範囲の確認結果 | 呼び出し元・関連テストの一覧 |

## 使い方

> **Copilotユーザー:** スラッシュコマンド `/cd-change-existing` で実行できます。以下はコマンドの内容を理解・カスタマイズするための参考情報です。

### パターンA: Agent Mode（推奨）

```
以下の変更仕様に基づいて、既存コードを改修してください。

変更対象: {{対象ファイルパス}}
変更仕様: {{変更内容の詳細}}

手順:
1. 対象ファイルと関連ファイル（{{関連パッケージパス}}）を読み取る
2. 既存のコーディングスタイル・例外処理パターンを把握する
3. 変更仕様に基づいてコードを改修する
4. 影響範囲（呼び出し元、関連するService/Repository）をプロジェクト内を検索して確認する
5. 影響を受けるファイルがあれば、必要な修正を提案する
6. 関連する既存テストの修正も提案する

制約:
- 変更仕様に記載されていない改善・リファクタリングは行わない
- 既存のコーディングスタイルを変更しない
- 既存コメントに記載された経緯・意図を尊重する
```

**カスタマイズのポイント:**

- `{{対象ファイルパス}}`: 改修対象のファイル（例: `src/main/java/com/example/app/order/OrderService.java`）
- `{{変更内容の詳細}}`: 変更仕様を具体的に記述（例: 「注文エンティティにdiscountAmountフィールドを追加し、合計金額計算に割引を反映」）
- `{{関連パッケージパス}}`: 影響確認の範囲（例: `com.example.app.order`パッケージ配下）

### パターンB: Chat / Inline（代替）

1メソッド内の小規模な修正にはInline補完が効率的。

```
{{対象ファイルパス}} の {{メソッド名}} メソッドに以下の変更を加えてください。
変更内容: {{変更内容}}
既存のスタイルに合わせてください。
```

## 人間が確認すべきこと

- [ ] 変更後のコードが変更仕様と整合しているか（仕様に対する過不足がないか）
- [ ] 既存テストが壊れていないか（`mvn test` を実行して確認）
- [ ] 影響範囲の変更漏れがないか（呼び出し元やDB定義の変更等）
- [ ] 既存のコーディングスタイル・命名規則と一致しているか

## よくある落とし穴

1. **仕様外の「改善」の混入**: AIが「良かれと思って」変更仕様に含まれないリファクタリングや最適化を行うことがある。差分を変更仕様と突き合わせ、仕様外の変更が含まれていないか確認する。
2. **意図的な実装の「修正」**: 既存コードのワークアラウンドや意図的な実装をAIがバグとみなして変更してしまうことがある。既存コードにコメントや経緯があれば、それを尊重しているか確認する。
3. **変更範囲の肥大化**: 1つの変更が多数のファイルに波及し、PRが巨大になることがある。1PRあたりの変更は小さく保ち、レビュー可能な単位に分割する。
