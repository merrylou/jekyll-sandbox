# UC06: 要件起点の修正箇所特定

## 目的

変更要件（機能仕様書・変更依頼書）を起点に、修正が必要なクラス・メソッドを逆引きで特定する。
「どこを直せばいいかわからない」状態を解消し、修正着手までの調査時間を短縮する。

## 利用タイミング

- 変更要件を受領した直後、実装着手前の調査フェーズ
- 既存コードベースが大規模で、修正箇所の見当がつかないとき
- 改修案件で影響範囲を事前に把握しておきたいとき

## Input / Output

| 区分 | 内容 | 形式 |
|------|------|------|
| Input | 変更要件の内容 | 機能仕様書・変更依頼書のテキスト、または自然言語での説明 |
| Input | 変更の種別 | 機能追加 / 仕様変更 / 廃止 |
| Input | 対象機能・画面名（任意） | 検索精度向上のための補助情報 |
| Output | 修正箇所一覧 | 必須 / 影響による変更 / 確認が必要 の3段階分類 |
| Output | 修正の推奨順序 | 依存関係を考慮した実施順 |

## 使い方

> **Copilotユーザー:** スラッシュコマンド `/cd-analyze-changepoint` で実行できます。以下はコマンドの内容を理解・カスタマイズするための参考情報です。

### パターンA: Agent Mode（推奨）

1. `@implementer` エージェントを呼び出す
2. `/cd-analyze-changepoint` プロンプトを実行する
3. 変更要件の内容・種別・対象機能を入力する
4. 出力された修正箇所一覧を確認する

### パターンB: Chat で部分的に調査

- 「この変更要件に関連するControllerはどれか」と質問する
- 特定のエントリーポイントからの呼び出しチェーンを確認する

## 人間が確認すべきこと

- [ ] AIが特定した修正箇所が実際に変更要件に関連しているか確認する
- [ ] 「確認が必要」とされた箇所を実際にコードを読んで判断する
- [ ] AIが見逃している修正箇所がないか、業務知識に基づいて確認する
- [ ] 新規作成が必要とされた箇所の妥当性を確認する

## よくある落とし穴

1. **AIの修正箇所リストを鵜呑みにする**: AIはコードの静的な依存関係は追えるが、業務ロジック上の暗黙の依存（例: 別テーブルのデータ整合性）は見逃すことがある。業務知識を持つメンバーと突き合わせること。
2. **調査と実装を混在させる**: このUCは修正箇所の「特定」に専念するもの。特定結果を確認してから実装（UC02 or UC08）に進むこと。
3. **対象機能の情報不足**: 変更要件だけでは検索キーワードが不足し、関連クラスを見逃す場合がある。画面名・API名・テーブル名など、できるだけ具体的な情報をAIに与えること。
