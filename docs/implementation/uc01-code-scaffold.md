# UC01: 設計書からのコード骨格生成

## 目的

詳細設計書からController / Service / Repository / DTO等の骨格コードを一括生成し、実装着手までのリードタイムを短縮する。
手作業でのボイラープレート作成を排除し、設計書との整合性を保った状態で実装を開始できる。

## 利用タイミング

- 詳細設計書（クラス設計・API仕様）の該当機能分がレビュー完了し確定した直後
- 新規機能の実装タスクに着手する時点

## Input / Output

| 区分 | 内容 | 形式 |
|------|------|------|
| Input | 詳細設計書（クラス設計・API仕様） | ドキュメント or 設計ファイル |
| Input | 既存コードベースのパターン | プロジェクト内のソースコード |
| Output | レイヤー別Javaソースコード | Controller / Service / Repository / DTO |
| Output | ビジネスロジック部分のTODOコメント | `// TODO: [設計書参照] xxxを実装` |

## 使い方

> **Copilotユーザー:** スラッシュコマンド `/cd-create-scaffold` で実行できます。以下はコマンドの内容を理解・カスタマイズするための参考情報です。

### パターンA: Agent Mode（推奨）

```
プロジェクトの既存パターン（{{既存の類似機能パッケージパス}}）を確認し、
以下の設計書に基づいてコード骨格を生成してください。

設計書: {{設計書ファイルパス}}
対象機能: {{機能名}}
パッケージ: {{ベースパッケージ}}

生成対象:
- Controller（リクエスト/レスポンスのマッピング）
- Service（インターフェースと実装クラス）
- Repository（Spring Data JPAのインターフェース）
- DTO（リクエスト/レスポンス/エンティティ）

制約:
- ビジネスロジック部分は // TODO: コメントで明示し、中身は実装しない
- 既存の例外体系（{{例外基底クラス名}}）に合わせる
- 命名規則は既存コードに準拠する
- バリデーションアノテーションは設計書の入力チェック仕様に基づく
```

**カスタマイズのポイント:**

- `{{既存の類似機能パッケージパス}}`: 参考にすべき既存機能のパス（例: `com.example.app.user`）
- `{{設計書ファイルパス}}`: 詳細設計書のファイルパスまたはURL
- `{{ベースパッケージ}}`: 生成先のパッケージ構成（例: `com.example.app.order`）
- `{{例外基底クラス名}}`: プロジェクト固有の例外クラス（例: `BusinessException`）

### パターンB: Chat / Inline（代替）

単一クラスの骨格だけ必要な場合に使用する。

```
以下のAPI仕様に基づいて、Spring Boot の RestController を1クラス生成してください。
既存の {{既存Controllerファイルパス}} のスタイルに合わせてください。

エンドポイント: {{HTTPメソッド}} {{パス}}
リクエスト: {{リクエスト仕様}}
レスポンス: {{レスポンス仕様}}
```

## 人間が確認すべきこと

- [ ] パッケージ構成がプロジェクトの規約通りになっているか
- [ ] アノテーション（`@Entity`, `@Valid`, `@Column`等）が設計書の仕様と一致しているか
- [ ] TODOコメントの箇所が明確で、実装すべき内容が分かるか
- [ ] 不要な依存（`pom.xml`への追加提案等）が含まれていないか
- [ ] 既存の基底クラス・インターフェースが正しく継承/実装されているか

## よくある落とし穴

1. **存在しないライブラリ・メソッドの使用**: AIが実在しないAPIやユーティリティメソッドを生成することがある。生成後は必ず `mvn compile` でビルドを通し、未解決の参照がないか確認する。
2. **セキュリティ実装の混入**: 認証・認可に関するアノテーションや実装をAIが独自に追加することがある。セキュリティ設計は別途レビューが必要なため、骨格生成時に含まれていたら一旦除去し、セキュリティ設計に基づいて改めて実装する。
3. **生成量過多によるレビュー困難**: 一度に大量のクラスを生成するとレビューが追いつかない。機能単位・レイヤー単位で分割して生成し、段階的にレビュー・コミットする。
