# UC08: 修正方針の策定

## 目的

修正対象と変更要件を入力に、複数の実装アプローチを比較した変更計画を策定する。
計画を承認してから実装に進むことで、手戻りリスクを低減する。

## 利用タイミング

- 修正方法が複数考えられ、どのアプローチが最適か判断したいとき
- 影響範囲が広い改修で、事前に計画を立ててからレビュー・承認を得たいとき
- UC06（修正箇所特定）の結果を受けて、具体的な実装方針を決めるとき

## Input / Output

| 区分 | 内容 | 形式 |
|------|------|------|
| Input | 修正対象のクラス名またはファイルパス | コード参照 |
| Input | 変更要件の内容 | 何をどう変えたいか |
| Input | 制約条件（任意） | 変更できない箇所、非機能要件等 |
| Output | アプローチ比較表 | 2〜3案を6観点（実装コスト・影響範囲・整合性・テスト容易性・リスク・要件充足度）で比較 |
| Output | 推奨案と選定理由 | 1案を推奨し根拠を明示 |
| Output | 変更計画サマリ | 修正ファイル一覧・変更概要・推奨実施順序 |

## 使い方

> **Copilotユーザー:** スラッシュコマンド `/cd-change-plan` で実行できます。以下はコマンドの内容を理解・カスタマイズするための参考情報です。

### パターンA: Agent Mode（推奨）

1. `@implementer` エージェントを呼び出す
2. `/cd-change-plan` プロンプトを実行する
3. 修正対象・変更要件・制約条件を入力する
4. 出力された変更計画を確認・承認する
5. 承認後、AIが推奨案に基づいて実装を開始する

### パターンB: Chat で方針を相談

- 「この修正をするなら、ServiceクラスとControllerクラスのどちらに手を入れるべきか」と相談する
- 特定のアプローチのメリット・デメリットを確認する

## 人間が確認すべきこと

- [ ] 比較されたアプローチが妥当か（他に検討すべき案がないか）
- [ ] 推奨案の選定理由が納得できるか
- [ ] 影響範囲の見積もりが漏れていないか
- [ ] テスト修正の見積もりが現実的か
- [ ] 計画を承認してから実装に進むこと（計画フェーズではコード変更しない）

## よくある落とし穴

1. **計画なしにいきなり実装に入る**: 影響範囲が広い改修を計画なしで始めると、途中で方針変更が必要になり大きな手戻りが発生する。このUCで事前に方針を固めることで防止する。
2. **AIの推奨案を無条件に採用する**: AIは既存コードの設計意図やチーム内の暗黙の制約を完全には理解できない。推奨案をたたき台として、チーム内で議論して最終決定すること。
3. **比較観点の偏り**: AIは実装コストや影響範囲など定量的な観点は得意だが、チーム内のスキルセットや今後の開発計画を踏まえた判断は苦手。人間の判断で補完すること。
