# UC05: レビュー前の品質セルフチェック

## 目的

PR提出前にAIを使って品質問題を事前検出し、レビューでの指摘事項を減らすことでレビュー往復回数を削減する。
レビュアーの負荷を下げ、レビューをより本質的な設計・ロジックの議論に集中させる。

## 利用タイミング

- 実装完了後、PRを作成する直前
- 自分で一通り見直した後、提出前の最終チェックとして実施する

## Input / Output

| 区分 | 内容 | 形式 |
|------|------|------|
| Input | 変更したコード（PRに含まれる差分） | 変更ファイル一覧 or git diff |
| Input | チーム固有のレビュー観点（任意） | コーディング規約 / チェックリスト |
| Output | 問題点の一覧 | 深刻度付き（Critical / High / Medium / Low） |
| Output | 改善提案 | 問題箇所ごとの具体的な修正案 |

## 使い方

> **Copilotユーザー:** スラッシュコマンド `/cd-check-review` で実行できます。以下はコマンドの内容を理解・カスタマイズするための参考情報です。

### パターンA: Chat（推奨）

```
以下の変更ファイルをレビューしてください。

変更ファイル:
- {{変更ファイルパス1}}
- {{変更ファイルパス2}}
- {{変更ファイルパス3}}

チェック観点:
1. バグ（NullPointerException、境界値、例外処理漏れ等）
2. セキュリティリスク（SQLインジェクション、XSS、認証認可の抜け等）
3. パフォーマンス問題（N+1クエリ、不要なDB呼び出し、メモリリーク等）
4. コーディング規約違反（{{コーディング規約の参照先}}）
5. 保守性（可読性、命名、責務の分離等）

出力形式:
| 深刻度 | ファイル | 行 | 問題内容 | 改善案 |
```

**カスタマイズのポイント:**

- `{{変更ファイルパス}}`: PRに含まれる変更ファイルを列挙する。`git diff --name-only` の出力を貼り付けると効率的
- `{{コーディング規約の参照先}}`: チーム固有の規約がある場合にファイルパスやURLを指定する（例: `docs/coding-standards.md`）
- チェック観点はプロジェクトの特性に合わせて追加・変更する（例: マルチテナント対応の確認、国際化対応等）

### パターンB: Agent Mode（代替）

複数ファイルの変更をまとめてチェックし、修正まで実施したい場合。

```
今回の変更差分（{{ブランチ名}} のコミット）を確認し、
以下の観点でセルフレビューを実施してください。

観点: バグ、セキュリティ、パフォーマンス、規約違反、保守性
問題が見つかった場合は、修正も実施してください。
Critical/Highの問題は必ず修正し、Medium以下は一覧として報告してください。
```

## 人間が確認すべきこと

- [ ] AIの指摘が妥当か確認する（偽陽性＝的外れな指摘が含まれることがある）
- [ ] Critical / High の指摘は内容を精査し、必ず対応する
- [ ] AIが見逃している観点がないか確認する（ビジネスロジックの正しさはAIでは判断困難）
- [ ] 指摘を反映した後、再度ビルド・テストが通ることを確認する

## よくある落とし穴

1. **AIセルフチェックによる人間レビューの省略**: AIのセルフチェックはあくまでレビュー前のフィルタリングであり、人間によるコードレビューの代替ではない。ビジネスロジックの妥当性、設計判断の適切さ、非機能要件の考慮は人間のレビューでしか確認できない。
2. **的外れな指摘への機械的な全対応**: AIはコードの文脈を完全に理解できないため、的外れな指摘や過剰な指摘をすることがある。各指摘を個別に判断し、妥当なもののみ対応する。判断に迷う場合はチームメンバーに相談する。
3. **チーム固有の規約・設計方針の見落とし**: AIはチーム固有のコーディング規約や設計方針を知らないため、それらに関する指摘は漏れる。チーム独自のチェックリストと併用し、AIチェックで網羅できない観点を補完する。
