# プレイブック整備の要件・背景

## 本ドキュメントの位置づけ

本リポジトリで整備するプレイブックの要件・背景・設計判断をまとめたものである。プレイブックの内容を改善する際の判断基準として参照する。

---

## 1. プロジェクト背景

### 組織の課題

本プロジェクトはSIer組織での以下の課題を解決するために発足した。

| 課題 | 詳細 |
|------|------|
| AI活用が点在 | 案件ごとに個別対応し、再利用可能な型がない。毎回ゼロからのAI活用準備に追加工数が発生。提案段階での競争力が低い |
| 品質基準の不在 | 統一されたレビュー観点・品質基準がなく、AI提案の採用基準が担当者に依存。品質が不安定 |
| 学習コストの高止まり | ベースとなるプロンプト・テンプレート・チェックリストがないため、エンジニアの立ち上がりに時間がかかる |

### 対象スコープ

| 項目 | 内容 | 理由 |
|------|------|------|
| 対象プロジェクト種別 | Webアプリ開発 x ウォーターフォール型 | 最も案件数が多い |
| 開始規模 | 既存コンテキストが存在する改修案件（小規模） | 新規開発より圧倒的に多く、効果を実感しやすい |
| 対象工程（初版） | 実装、単体テスト | 最優先。段階的に詳細設計等へ拡大 |
| 対象AIツール | GitHub Copilot（Agent Mode含む） | 現時点の社内標準 |
| 開発規模 | 10〜30名チーム | 組織の標準的なチーム規模 |
| 技術スタック（具体例） | Java / Spring Boot | 主要案件の技術スタック |

## 2. プレイブックで実現したいこと

### 基本コンセプト

**工程別にユースケースを整理し、各ユースケースの具体的な使い方をガイドする。**

プレイブックの構造イメージ：

```
工程（実装、UT等）
  └── ユースケース一覧（概要レベル）
        └── ユースケース詳細
              ├── 目的
              ├── 利用タイミング
              ├── Input / Output
              ├── 人間が確認すべきこと
              └── 使い方（プロンプト例 + カスタマイズ方針）
```

### プロンプト例に対する考え方

プロンプト例はそのまま利用できることは稀である。そのため、**カスタマイズ方針**を合わせてガイドし、利用者が自チームのプロジェクトに合わせて調整できるようにする。

Agent Modeの場合、従来のコピペ型テンプレートとは異なり、タスク指示型のプロンプト（ファイル参照、既存パターンへの準拠指定等）が中心となる。

### 進め方の方針

| 方針 | 理由 |
|------|------|
| 小さく始めて改善を繰り返す | 最初から難しいことを考えすぎても進まない |
| 実際に利用してフィードバックを得る | 作ったものを使ってみないと分からない |
| 利用者が見てくれるボリューム感を維持 | 大規模なものを作りすぎると利用者が見ない・使われない |

## 3. 設計判断の記録

### ファイル分割の考え方

| 分割軸 | 採用 | 理由 |
|--------|------|------|
| 工程別（実装 / UT） | 採用 | 利用者は自分の担当工程のものだけ見たい |
| ユースケース別（1ファイル1UC） | 採用 | 必要なものだけ読める。個別に改善・追加しやすい |
| 共通ドキュメント（common/） | 採用 | Copilotモード解説、レビューチェックリスト等は工程横断で重複を避ける |
| スキルレベル別 | 不採用 | メンテナンスコストが倍増する。各UCが初心者・経験者双方に対応する構成で代替 |
| 技術スタック別 | 不採用（初版） | 初版はJava/Spring Bootに絞る。他スタック追加時はディレクトリで対応可能 |

### ボリューム感の設計

| 項目 | 設計値 | 理由 |
|------|--------|------|
| ユースケース数 | 5-8件/工程 | 主要なものを一通りカバーしつつ、読める量に収める。初版では実装8件・UT5件 |
| ユースケース詳細 | 60-100行/ファイル | 印刷2-3ページ相当。5分で読める量 |
| 人間が確認すべきこと | 最大5項目 | 多すぎると読まれない |
| よくある落とし穴 | 最大3項目 | 同上 |

### 将来検討: 製造工程のINPUTとなる設計成果物の再定義

AI活用が進むと、従来の詳細設計成果物の中で不要になるものが出てくる可能性がある。特にプログラム設計書（クラス図、メソッド定義、処理フロー等）は、AIが「基本設計 + API仕様 + 既存コード」から直接コードを生成できる領域であり、人間が事前に書く成果物としての必要性が薄れる。

| 成果物 | 従来の位置づけ | AI活用後の見通し |
|--------|---------------|-----------------|
| 基本設計書 | 引き続き必要 | 変更なし。AIへのINPUTとしても重要 |
| API仕様書 | 引き続き必要 | 変更なし。AIによるコード生成の主要INPUT |
| DB設計書（テーブル定義等） | 引き続き必要 | 変更なし。AIによるEntity/Repository生成のINPUT |
| 画面項目定義書 | 引き続き必要 | 変更なし |
| プログラム設計書 | 製造の前提として事前作成 | AIがコードを生成できる領域と重複。事前作成の必要性が低下 |

**初版では対応しない理由:**
- 顧客との契約でプログラム設計書が納品物に含まれている場合、契約変更の交渉が先に必要
- 設計レビュー完了を製造着手条件とするゲート運用の再定義が必要
- AI生成コードの設計責任の所在を組織として整理する必要がある

**将来の方向性（案）:**
プログラム設計書を「事前に書くもの」から「コード実装後に逆生成するもの」に変える段階的アプローチが考えられる。これにより契約上の納品物を満たしつつ、実質的な設計工数を削減できる。本プレイブックが定着し、AI活用の実績が蓄積された段階で具体的な検討を開始する。

---

### 初版で意図的に省略したもの

| 省略項目 | 理由 | 将来の扱い |
|---------|------|-----------|
| Good/Bad出力例 | ボリューム増大を避ける。「人間が確認すべきこと」で代替 | フィードバックに応じて追加検討 |
| 工数削減の数値目標 | 根拠なき数値は危険（PM視点レビューでの指摘） | 実績データが溜まってから記載 |
| CI/CD統合ガイド | 初版では手動運用に集中 | Phase 2以降で整備 |
| 他工程（設計、テスト計画等） | 全工程を一度にやると進まない | 実装・UTが安定してから拡大 |
