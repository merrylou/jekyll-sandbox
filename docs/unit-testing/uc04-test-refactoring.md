# UC04: 既存テストコードの改善

## 目的

レガシーなテストコード（JUnit4、手動Mock、保守性の低いテスト等）を現行のテスト基盤に合わせて改善する。テストの可読性・保守性を向上させつつ、既存のテストカバレッジを維持する。

## 利用タイミング

- 改修案件で既存テストに手を入れる必要がある時
- テスト基盤のバージョンアップ時（JUnit4からJUnit5への移行等）
- テストコードの可読性が低く、保守コストが高い時

## Input / Output

| 区分   | 項目                 | 説明                                                 |
| ------ | -------------------- | ---------------------------------------------------- |
| Input  | 改善対象の既存テスト | JUnit4等で書かれた既存テストクラス                   |
| Input  | 移行先のテスト基盤   | JUnit5 + Mockito + AssertJ等の目標構成               |
| Output | 改善後のテストコード | 移行先基盤に合わせたテストクラス                     |

## 使い方

> **Copilotユーザー:** スラッシュコマンド `/ut-change-test` で実行できます。以下はコマンドの内容を理解・カスタマイズするための参考情報です。

### パターンA: Agent Mode（推奨）

```
{{既存テストファイルパス}} を読み取り、
以下の方針でテストコードを改善してください。

【移行方針】
- JUnit4 → JUnit5 に移行する
  - @Test: org.junit.Test → org.junit.jupiter.api.Test
  - @Before/@After → @BeforeEach/@AfterEach
  - @BeforeClass/@AfterClass → @BeforeAll/@AfterAll
  - @RunWith → @ExtendWith に変更
  - @Rule ExpectedException → assertThrows に変更
- Mock: 手動Mock / PowerMock → Mockito + @ExtendWith(MockitoExtension.class)
- アサーション: JUnit Assert / Hamcrest → AssertJ に統一
- テストメソッド名: {{日本語命名規則「正常系_〇〇の場合_〇〇を返す」}}

【制約】
- テストの意図・カバレッジは維持すること（テストケースの追加・削除はしない）
- 不要になったimport文は削除すること
- {{チームのテスト規約ファイルパス}} があれば参照すること
```

**カスタマイズのポイント:**

- `{{既存テストファイルパス}}`: 改善対象のテストクラスを指定
- `{{日本語命名規則「正常系_〇〇の場合_〇〇を返す」}}`: チームの命名規約に合わせて変更。英語なら `shouldXxx_whenYyy` 等
- `{{チームのテスト規約ファイルパス}}`: テスト規約ドキュメントがあれば参照先を指定。なければこの行を削除
- 移行元がJUnit5で可読性改善のみの場合は、移行方針のJUnit4関連の記述を削除し、アサーション統一やメソッド名改善に絞る

### パターンB: Chat

特定のテストメソッド1つだけを書き換えたい場合に使用する。

```
以下のJUnit4テストメソッドをJUnit5 + Mockito + AssertJに書き換えてください。
テストの意図は変えないでください。

{{テストメソッドのコードを貼り付け}}
```

## 人間が確認すべきこと

- [ ] 移行前と同じテストケース（同じ条件・同じ期待値）がカバーされているか
- [ ] テスト実行が全件パスするか
- [ ] 不要になったimport文や依存ライブラリが残っていないか
- [ ] テストの意図が読み取りやすくなっているか（可読性が向上しているか）

## よくある落とし穴

1. **AIが移行ついでにテストロジックを変更することがある**: フレームワーク移行に加えて、テスト条件や期待値まで変えてしまうケースがある。移行前後でテスト件数・テスト条件が一致しているか差分を確認する。
2. **カスタムTestRunnerやRule等を正しく変換できないことがある**: `@RunWith` でカスタムRunnerを使っている場合や、独自の `@Rule` を使っている場合、AIが適切な `@ExtendWith` への変換を行えないことがある。こうしたケースは手動で対応する。
3. **大量のテストを一度に変換するとレビューが困難になる**: 数十ファイルを一括変換すると差分の確認が追いつかない。ファイル単位で段階的に変換し、変換のたびにテスト実行して動作確認する。
