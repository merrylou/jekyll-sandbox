# UC02: テストデータ生成

## 目的

テストケースに必要なテストデータ（テストフィクスチャ、SQL、JSON等）を効率的に生成する。手作業でのデータ準備工数を削減し、網羅性の高いテストデータを短時間で揃える。

## 利用タイミング

テストケースの設計が完了し、具体的なテストデータの準備に着手するタイミング。テーブル定義やDTO定義が確定していることが前提。

## Input / Output

| 区分   | 項目                          | 説明                                              |
| ------ | ----------------------------- | ------------------------------------------------- |
| Input  | テーブル定義またはDTO定義     | DDL、Entityクラス、またはER図から抽出した定義      |
| Input  | テストシナリオ                | 正常系・異常系・境界値のどのパターンが必要か       |
| Output | テストデータ（SQL/JSON/Java） | INSERT文、JSONファイル、Javaのテストフィクスチャ等 |

## 使い方

> **Copilotユーザー:** スラッシュコマンド `/ut-create-testdata` で実行できます。以下はコマンドの内容を理解・カスタマイズするための参考情報です。

### パターンA: Chat（推奨）

```
以下のテーブル定義に対して、テストデータをSQL INSERT文で生成してください。

【テーブル定義】
{{DDLまたはEntityクラスを貼り付け}}

【生成条件】
- 正常系: {{N}}件（代表的なパターンを網羅）
- 境界値: 文字列カラムの最大長、数値カラムの上限/下限、日付の境界
- 異常系: NULL許容カラムのNULL、特殊文字（'、"、<、>）を含むデータ
- メールアドレスは test+xxx@example.com 形式を使用
- パスワードはダミーハッシュ値（$2a$10$DUMMY...）を使用
- 個人情報に似たデータは避け、明らかに架空のデータにすること

【出力形式】
- {{SQL INSERT文 / JSON / Javaテストフィクスチャ}}
```

**カスタマイズのポイント:**

- `{{DDLまたはEntityクラスを貼り付け}}`: テーブル定義をDDLで貼るか、JPA Entityクラスを貼るか選択
- `{{N}}`: 正常系データの件数。通常3〜5件で十分
- `{{SQL INSERT文 / JSON / Javaテストフィクスチャ}}`: 出力形式を用途に合わせて選択。Spring Boot Testの `@Sql` で使うならSQL、REST APIテストならJSON、Builder/Factoryパターンで使うならJava
- マスキングルールがある場合は生成条件に追加する

### パターンB: Agent Mode

`@Sql` アノテーション用のSQLファイルをテストクラスと合わせて生成したい場合に使用する。

```
{{テスト対象ファイルパス}} と {{テーブル定義ファイルパス}} を読み取り、
テストクラスで使用するテストデータSQLファイルを生成してください。

- src/test/resources/sql/ 配下に配置するINSERT文を生成
- テストクラスの各テストメソッドに対応するデータを用意
- @Sql アノテーションで読み込む前提の構成にする
```

## 人間が確認すべきこと

- [ ] 本番データや実在する個人情報に似たデータが含まれていないか
- [ ] テストシナリオの条件（正常系/異常系/境界値）を正しく満たしているか
- [ ] データ間の整合性（FK制約、一意制約等）が保たれているか

## よくある落とし穴

1. **AIが「リアルっぽい」データを生成し実在の情報に近づくことがある**: 住所や電話番号がリアルなものになりやすい。「明らかに架空のデータにすること」をプロンプトに明示する。生成後も目視で確認する。
2. **テストデータの量が多すぎるとテストが遅くなる**: 特にINSERT文を大量に生成すると、テスト実行時間が膨らむ。各シナリオに必要最小限のデータに絞る。
3. **FK制約や一意制約を考慮していないデータが生成される**: AIは単一テーブルのデータを独立して生成する傾向がある。関連テーブルの定義も合わせて提示し、制約を満たすデータを生成するよう指示する。
