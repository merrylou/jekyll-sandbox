# UC05: テスト仕様設計

## 目的

実装コードや設計書からテスト観点を導出し、テスト仕様書（UT観点一覧）を作成する。
テスト設計を先行させることで、テスト実装の方向性を明確にし、観点の漏れを防ぐ。

## 利用タイミング

- 新規機能のテスト実装に着手する前（テスト設計フェーズ）
- テストレビュー時に観点の網羅性を確認したいとき
- カバレッジ目標を達成するためのテスト計画を立てるとき

## Input / Output

| 区分 | 内容 | 形式 |
|------|------|------|
| Input | テスト対象のクラスまたは設計書 | 実装ファイル or 設計ドキュメント |
| Input | カバレッジ目標（任意） | ブランチカバレッジ等の目標値 |
| Output | テスト観点一覧 | テストケースの条件・期待結果の表 |
| Output | パラメータ化テスト候補 | @ParameterizedTest の適用候補 |

## 使い方

> **Copilotユーザー:** スラッシュコマンド `/ut-design-testspec` で実行できます。以下はコマンドの内容を理解・カスタマイズするための参考情報です。

### パターンA: Agent Mode（推奨）

1. `@tester` エージェントを呼び出す
2. `/ut-design-testspec` プロンプトを実行する
3. テスト対象クラス・カバレッジ目標を入力する
4. 生成されたテスト観点一覧を確認・修正する

### パターンB: Chat で部分的に観点を相談

- 「このメソッドの境界値テストにはどんなケースが必要か」と質問する
- 特定の分岐条件に対するデシジョンテーブルを相談する

## 人間が確認すべきこと

- [ ] テスト観点が網羅されているか（同値分割・境界値・異常系・状態遷移）
- [ ] コード上の分岐だけでなく、業務仕様上重要なケースが含まれているか
- [ ] @ParameterizedTest候補が適切か（パラメータ化により可読性が下がっていないか）
- [ ] 観点数がカバレッジ目標の達成に十分か

## よくある落とし穴

1. **AIが正常系に偏り、異常系・境界値が不足する**: AIは正常系のテストケースを多く生成する傾向がある。「異常系・境界値を重点的に」と指示することで偏りを補正する。
2. **デシジョンテーブルの組み合わせ爆発**: 条件数が多いとAIが全組み合わせを列挙し、テストケースが膨大になる。条件数4以上は代表的な組み合わせに絞るよう指示する。
3. **観点がテスト実装に落とし込めない粒度**: 抽象的な観点だけでは実装時に迷う。出力形式に「入力値・前提条件」「期待結果」列を必須にすることで、テスト実装に直結する粒度にする。
