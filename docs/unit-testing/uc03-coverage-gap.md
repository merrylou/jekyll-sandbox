# UC03: カバレッジギャップの特定と補完

## 目的

テストカバレッジの不足箇所を特定し、不足分のテストを効率的に追加する。カバレッジ目標達成までの作業を短縮する。

## 利用タイミング

- 初回テスト生成後にカバレッジを計測し、目標値（例: ブランチカバレッジ80%）に達していない時
- 既存テストのカバレッジ改善タスクに着手する時

## Input / Output

| 区分   | 項目                     | 説明                                             |
| ------ | ------------------------ | ------------------------------------------------ |
| Input  | テスト対象のソースコード | カバレッジを改善したい実装クラス                 |
| Input  | 既存のテストコード       | 現時点のテストクラス                             |
| Input  | カバレッジレポート       | JaCoCo等のレポート（あれば。なくても分析は可能） |
| Output | 追加テストコード         | 不足分のテストメソッド                           |
| Output | 補完テストケース一覧     | 追加したテストの観点・条件の一覧                 |

## 使い方

> **Copilotユーザー:** スラッシュコマンド `/ut-analyze-coverage` で実行できます。以下はコマンドの内容を理解・カスタマイズするための参考情報です。

### パターンA: Agent Mode（推奨）

```
以下のファイルを読み取り、テストカバレッジのギャップを分析してください。

【テスト対象】{{テスト対象ファイルパス}}
【既存テスト】{{既存テストファイルパス}}

【指示】
1. テスト対象クラスの全分岐・条件を洗い出す
2. 既存テストでカバーされていない分岐・条件を特定する
3. 不足分のテストメソッドを追加生成する

【制約】
- 既存テストのスタイル（命名規則、アサーション方式、Mockの使い方）に合わせる
- 既存テストファイルに追記する形で出力する
- カバレッジ目標: {{ブランチカバレッジ80%}}
- 重点的にカバーしたい観点: {{例外系 / 分岐条件 / null処理}}
```

**カスタマイズのポイント:**

- `{{テスト対象ファイルパス}}`: カバレッジを改善したいクラスを指定
- `{{既存テストファイルパス}}`: 既存テストクラスを指定。AIが既存のスタイルを読み取って合わせる
- `{{ブランチカバレッジ80%}}`: チームのカバレッジ目標値に合わせて変更
- `{{例外系 / 分岐条件 / null処理}}`: 特に重点的にカバーしたい観点を指定。省略可
- JaCoCoレポートがある場合は、未カバー箇所を要約して追加情報として伝えると精度が上がる

### パターンB: Chat

特定メソッドの未テスト分岐を確認したい場合に使用する。

```
以下のメソッドで、テストされていない分岐・条件を教えてください。

【実装】
{{メソッドのコードを貼り付け}}

【既存テスト】
{{既存テストメソッドを貼り付け}}
```

## 人間が確認すべきこと

- [ ] 追加テストが既存テストと重複していないか（同じ条件を別の書き方でテストしていないか）
- [ ] カバレッジ改善だけでなく、テストとして意味があるか（形だけのテストでないか）
- [ ] 追加テストを実行してパスするか
- [ ] 既存テストが壊れていないか（副作用がないか）

## よくある落とし穴

1. **空のアサーションでカバレッジだけ稼ぐテストが生成される**: `assertNotNull(result)` だけ、または `assertDoesNotThrow(() -> ...)` だけのテストはカバレッジには貢献するが品質保証にならない。全テストメソッドに具体的な値のアサーションがあるか確認する。
2. **JaCoCoレポートをAIに直接読ませるのが難しい**: HTML/XMLのレポートをそのまま渡しても精度が低い場合がある。未カバー箇所を「〇〇メソッドのelse分岐が未カバー」のように人間が要約して伝える方が精度が上がる。
3. **privateメソッドを直接テストしようとする**: AIがリフレクション等でprivateメソッドを直接テストするコードを生成することがある。publicメソッド経由で間接的にカバーする方針をプロンプトで明示する。
