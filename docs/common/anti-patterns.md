# AI活用の共通アンチパターン集

## 概要

AI活用で陥りやすい共通のアンチパターンと対策をまとめる。
各項目は簡潔に記載しているため、詳細はチーム内で議論・補足すること。

---

## アンチパターン一覧

| アンチパターン | 問題 | 対策 |
|---|---|---|
| **AI出力をそのままマージ** | AI生成コードを理解せずにPRに含める。障害時に原因調査ができない | 自分の言葉で説明できないコードはマージしない。差分を1行ずつ確認する。`/cd-check-review` でセルフレビューを実施 |
| **巨大PR一括生成** | Agent Modeで大量のコードを一度に生成し、レビュー不能なPRを作る | 機能単位・ファイル単位でPRを分割する。生成コミットと修正コミットを分ける。`/cd-change-existing` で機能単位にスコープを区切って実行 |
| **AIレビューで人間レビュー省略** | AIのセルフチェック結果のみでマージする。AIは文脈や業務仕様を十分に理解できない | AIレビューはフィルタリング目的で活用する。人間レビューは省略しない。`/cd-check-review` の結果は参考情報として扱う |
| **本番データをAIに入力** | 動作確認のつもりで本番データや顧客情報をプロンプトに含める | 必ず架空データ・ダミーデータを使用する。マスキングルールを事前に決めておく。`/ut-create-testdata` でダミーデータを生成 |
| **仕様外の改善を採用** | AIが提案した「良さそうな改善」を仕様確認なしに採用する | 変更は仕様に紐づけること。仕様外の提案は別チケットとして起票する。`/cd-analyze-spec` で仕様との整合性を確認 |
| **セキュリティ実装をAI任せ** | 認証・認可・暗号化等のセキュリティ関連をAI生成のまま使う | セキュリティ関連は必ず専門知識を持つ人間がレビューする。`/cd-check-review` のセキュリティ観点を活用 |
| **テストの数だけ追求** | カバレッジ数値のために中身の薄いテストを量産する | アサーションの質を確認する。仕様に基づくテストになっているか検証する。`/ut-design-testspec` でテスト設計から着手 |
| **AI依存でスキル劣化** | 常にAIに書かせることで、自分でコードを書く・読む力が低下する | AIなしでもコードを書ける状態を維持する。AIの提案を「なぜそうなるか」理解する習慣をつける。`/cd-analyze-codebase` でコード理解を深める |

---

これらはAI活用の初期段階で特に起きやすい。チーム内で定期的に振り返り、該当していないか確認すること。
