# POC実施計画: spring-petclinicによるAI駆動開発プレイブック検証

## 1. 目的

本プレイブック（aidd-guide）が実際の開発作業で機能するかを、spring-petclinicを題材にしたPOCで検証する。

### 検証軸

| 軸 | 問い | 判定方法 |
|----|------|---------|
| 有効性 | プレイブックのUCが実コードベースで機能するか | 7UC以上（対象8UC中）をend-to-endで実施完了 |
| 生産性 | AI駆動開発が従来手法と比較して効率的か | フェーズごとの所要時間・手動修正回数を記録 |
| 品質 | ガードレールが品質を担保するか | テストカバレッジ、アンチパターン発生数、レビュー結果 |

---

## 2. 題材と改修シナリオ

### 題材: spring-petclinic

| 項目 | 内容 |
|------|------|
| GitHub | https://github.com/spring-projects/spring-petclinic |
| 技術スタック | Spring Boot + Spring Data JPA + Thymeleaf + H2 |
| 選定理由 | Spring公式リファレンス、小規模で全体把握可能、既存テストあり（[詳細](spring-boot-oss-candidates.md)） |

### 改修シナリオ: ペットのワクチン接種記録管理

飼い主のペットに対するワクチン接種履歴を記録・参照できる機能を追加する。

**追加するもの:**

- `Vaccination` エンティティ（id, pet(FK), vaccineType, vaccinationDate, veterinarian(FK to Vet), notes）
- ペット詳細画面からワクチン接種履歴の一覧表示
- ワクチン接種記録の新規登録フォーム
- バリデーション（vaccineType必須、未来日不可、pet必須）

**シナリオ選定理由:** 新規エンティティ追加（UC01）＋ 既存コード修正（UC02）＋ テスト生成（UT UC01/02/03）を自然にカバーでき、既存のVisit（診察記録）と構造が類似するためパターン参照も容易。

---

## 3. 前提条件

- 実施者: 1名（Java/Spring Boot開発経験あり）
- ツール: GitHub Copilot（Agent Mode / Chat / Inline Completion）
- 環境: ローカル開発環境（JDK 17+, Maven）
- .github/ 配下のCopilotネイティブ設定をpetclinicリポジトリに配置して使用

### Phase 2: スキャフォールド生成（Day 1 / 約2〜3時間）

**対応UC:** 実装 UC01（設計書からのコード骨格生成）

**目的:** Vaccination機能のクラス骨格を生成する

**事前準備:** 簡易設計書を用意する（以下の情報を含むテキスト）

```
【エンティティ定義】
- Vaccination: id(Integer), pet(Pet, FK), vaccineType(String, 必須),
  vaccinationDate(LocalDate, 必須, 未来日不可), veterinarian(Vet, FK), notes(String)

【URL設計】
- GET  /owners/{ownerId}/pets/{petId}/vaccinations     → 一覧表示
- GET  /owners/{ownerId}/pets/{petId}/vaccinations/new  → 登録フォーム表示
- POST /owners/{ownerId}/pets/{petId}/vaccinations/new  → 登録処理

【画面】
- ワクチン接種履歴テーブル（vaccineType, date, vet, notes）
- 登録フォーム（vaccineType, date, vet選択, notes）
```

**手順:**

1. `@implementer` エージェントを選択する
2. `/cd-create-scaffold` スラッシュコマンドを実行する

   - 既存パターン参照: 「既存のVisit実装（Visit.java, VisitController, VisitRepository）と同じ構成で」と明示する
3. 生成されたファイルを確認する
4. `mvn compile` でコンパイルが通ることを確認する
5. コミットする

**確認チェックリスト:**

- [ ] パッケージ構成がpetclinicの既存パターンと一致しているか
- [ ] JPA アノテーション（@Entity, @Table, @ManyToOne 等）が設計通りか
- [ ] TODOコメントで未実装箇所が明示されているか
- [ ] 仕様にない追加ライブラリや依存が含まれていないか
- [ ] 既存の基底クラス（BaseEntity等）を正しく利用しているか
- [ ] `mvn compile` が通るか

### Phase 5: テスト生成 + テストデータ作成（Day 2〜3 / 約3〜4時間）

**対応UC:** UT UC01（テスト自動生成）+ UT UC02（テストデータ生成）

**目的:** Vaccination関連の単体テストを生成し、テストデータを整備する

**手順（UC01 — テスト生成）:**

1. `@tester` エージェントを選択する
2. `/ut-create-test` スラッシュコマンドを実行する
   - 対象: VaccinationController, Vaccination エンティティ, VaccinationRepository
   - 既存テストを参照パターンとして指定する（「既存の OwnerControllerTests / PetControllerTests と同じスタイルで」）
3. 生成されたテストを確認する
4. `mvn test` でテストが通ることを確認する

**手順（UC02 — テストデータ）:**

1. `/ut-create-testdata` スラッシュコマンドを実行する
   - 対象: Vaccinationエンティティのフィールド定義 + バリデーションルール
   - 生成対象: 正常系データ、境界値データ（空文字vaccineType、未来日、null pet）、異常系データ
2. 生成されたテストデータを確認する

**確認チェックリスト（テスト生成）:**

- [ ] 正常系・異常系・境界値のテストケースが含まれているか
- [ ] assertの期待値が仕様と一致しているか（AIの期待値を鵜呑みにしない）
- [ ] テスト命名が規約に従っているか（日本語: `正常系_〇〇の場合_〇〇を返す`）

- [ ] Mockの設定が適切か（過度なMockで実際の動作と乖離していないか）

**確認チェックリスト（テストデータ）:**

- [ ] 個人情報・本番データが含まれていないか
- [ ] FK制約等のDB制約を満たしているか

## 8. 本POCでカバーしないUC

以下のUCは本シナリオでの自然な検証機会がないため、スコープ外とする。自然な機会があれば併せて検証する。

- **実装UC06〜08（修正箇所特定・エラーログ解析・修正方針策定）**: Vaccination追加は新規機能であり、「既存コードの修正箇所逆引き」「エラーログ解析」「複数アプローチ比較」のシナリオが発生しにくい
- **UT UC04（テストリファクタリング）**: petclinicは既にJUnit5を採用しており、JUnit4→5移行シナリオが存在しない
- **UT UC05（テスト仕様設計）**: Phase 5でテスト生成を直接実施するため、テスト設計フェーズを明示的に設けていない
