# 開発者視点レビュー — 2026-04-02 #01

## レビュー対象

リポジトリ全体（README.md、docs/配下の全ドキュメント）。初版ドラフト時点。

## 総合評価: 4 / 5（良い）

実用性が高く、現場で「開いて使える」レベルに到達している。初版としては十分な完成度。ただし、初学者が「最初の一歩」を踏み出す際の心理的・実務的ハードルにもう少し配慮があると、組織全体への浸透がスムーズになる。

## 良い点

### 1. ユースケース単位の分割が実務にフィットしている

「設計書からの骨格生成」「既存コードの改修」「カバレッジギャップ補完」など、SIerの改修案件で日常的に発生するタスクがそのままUC名になっている。「今から何をやるか」でドキュメントを引ける構造は、辞書的に使えるので継続利用につながりやすい。

### 2. プロンプト例の「カスタマイズのポイント」が実践的

`{{}}`でプレースホルダを明示し、その直下に「何を入れるか」の具体例が書かれている。単なるコピペテンプレートではなく「自分のプロジェクトに合わせて何を変えればいいか」が分かるため、カスタマイズの負荷が低い。特にAgent Modeのプロンプトで「制約」を明記する構成は、AI出力の暴走を防ぐうえで非常に有用。

### 3. 「よくある落とし穴」が具体的で、実際にハマりそうなものばかり

「仕様外の改善の混入」「AIが存在しないライブラリを使う」「空アサーションでカバレッジだけ稼ぐ」など、Copilotを使い始めた段階で確実に遭遇する問題が列挙されている。各UC最大3項目という制限も、読む気を削がない適切なボリューム。

### 4. 「人間が確認すべきこと」がチェックリスト形式で実用的

各UCの確認項目が4〜5個に絞られており、PRセルフチェック時に実際に使える粒度になっている。「全てOKでなくても構わない。NGの理由を把握した上でレビューに出すこと」（review-checklist.md）という記述も現実的で好感が持てる。

### 5. 「どのUCから始めるか」のガイドが初心者に親切

実装工程index.mdの「AI活用が初めての方はUC04（コードベースの理解）から。コードを変更しないため安全に試せる」という導線は、初めてCopilotを使う開発者の不安を下げる良い設計。

## 懸念・改善提案（優先度順）

### 1.【高】AI活用に対する心理的ハードルへの配慮が薄い

**問題:** アンチパターン集に「AI依存でスキル劣化」の項目があるが、対策が「AIなしでもコードを書ける状態を維持する」と一言で片づけられており、開発者の不安に正面から向き合えていない。「AIに書かせたコードだとレビューで指摘されるのでは」「AIを使うのは手抜きだと思われるのでは」という心理的抵抗は、現場では根強い。

**提案:** 共通ガイドまたはREADMEに「AIを使うことはスキルの代替ではなく増幅である」というメッセージを明示する。具体的には:
- AIの出力を読み解き、適切に判断・修正すること自体が高いスキルを要求される行為であること
- レビューではコードの品質を見るのであり、書いた手段（手書き/AI）は問わないというチームの姿勢を明文化すること
- AIの提案を「なぜそう書くのか」理解する過程で、むしろ学びが加速する側面があること

### 2.【高】最初の一歩の具体性が不足している

**問題:** README.mdの「使い方」は3ステップで簡潔だが、Copilot自体のセットアップや基本操作が分かっている前提になっている。経験2年程度の開発者がGitHub Copilotを使い始めたばかりの状態で、いきなりAgent Modeのプロンプトを書けるかというと難しい。

**提案:** 以下のいずれかを追加する:
- 「クイックスタート」セクション（5分で試せる最小手順）: 具体的に「VS CodeでCopilot Chatを開き、`#file:OrderService.java` このクラスの処理フローを説明して、と入力する」レベルの手順
- Copilotの基本操作（Chatパネルの開き方、Agent Modeへの切り替え方、`@workspace`の使い方）への外部リンクまたは最低限の説明

### 3.【中】Agent Modeの成功/失敗の判断基準が不明確

**問題:** Agent Modeのプロンプト例は充実しているが、「AIの出力がうまくいかなかった場合にどうするか」の記述がない。実際にはAgent Modeの最初の出力が期待通りになることは少なく、プロンプトの追加指示や手動修正が必要になる場面が多い。初回で失敗すると「使えない」と判断して離脱するリスクがある。

**提案:** 各UCまたは共通ガイドに「うまくいかない時の対処法」を簡潔に追加する:
- 出力が意図と違う場合は「追加指示で軌道修正」する（「〇〇の部分は既存の△△に合わせて修正して」）
- スコープを狭める（「まずControllerだけ生成して」）
- Agent Modeで駄目ならChat + 手動編集に切り替える

### 4.【中】READMEのファイル名が実際のファイル名と不一致

**問題:** README.mdのドキュメント構成ツリーに記載されているファイル名（`uc01-scaffold.md`、`uc02-modify.md`等）と、実際のファイル名（`uc01-code-scaffold.md`、`uc02-existing-code-mod.md`等）が一致していない。ドキュメントを信じてファイルを探すと見つからない。

**提案:** README.mdのファイルツリーを実際のファイル名に合わせて修正する。

### 5.【低】効果の目安がないため上長への説明材料にならない

**問題:** requirements.mdに「工数削減の数値目標は根拠なき数値は危険」として意図的に省略されている。判断としては正しいが、現場の開発者が「これを使いたい」と上長やPMに説明する際、効果の目安がないと説得力に欠ける。

**提案:** 数値目標ではなく「効果を実感しやすい最初のユースケース」として、例えば「テスト自動生成（UT UC01）で初稿作成の時間が体感で半分以下になった」のような定性的な声のサンプルや、「まず1つのUCを1週間試して、チームで効果を振り返る」という導入ステップを追加する。

## 開発者視点での一言コメント

正直に言うと、こういうドキュメントは「作って終わり」になりがちで期待していなかったが、このプレイブックは「今日の改修タスクで実際に使える」レベルになっていて驚いた。特にプロンプトの「制約」欄で「仕様外の改善をするな」「既存スタイルを変えるな」と明示する構成は、AIの「余計なお世話」を防ぐのに即効性がある。一方で、Agent Modeで一発でうまくいかなかった時に「次にどうすればいいか」が書かれていないのが惜しい。Copilot初心者が最初の壁で挫折しないよう、「うまくいかない時はスコープを狭めてリトライ」という一言があるだけで、このガイドの定着率は大きく変わると思う。
